# Follow-ups

Tracker for known polish items that surfaced during phase 9.5 (publishing) and should be addressed before the first `1.0.0` release — or in a `1.0.x` patch right after.

## Visual polish

- **Item label alignment. ✅ WON'T FIX (by design).** Labels next to the icon column can look slightly off-baseline in normal mode (`docs/screenshots/1-modes.png`, `3-children.png`, `5-active-root.png`, `6-active-deep.png`). The addon sets `vertical-align: middle` on the label/suffix because those live in the light DOM where the item is block-level (the flex row is internal to the shadow `::part(content)`); that `middle` is required and correct. Going further — a fixed 1–2px nudge/padding — would be fragile: the actual optical alignment depends on which prefix icon is used and its size, so a built-in tweak that looks right for `VaadinIcon`s at Lumo's default size could look wrong for a larger/custom prefix. Left as-is; apps that need pixel-exact alignment for their specific icon set can override `::part(content)` / the label in their own CSS.

- **Rail icon styling when an ancestor is current. ✅ RESOLVED via docs (not a code default).** When a sub-sub item is the current route (e.g. `admin/users/active`), the rail-side root icon (Admin) inherits the full `current` styling — same highlight as if Admin itself were the current page (`docs/screenshots/6-active-deep.png`). This is **Vaadin's stock side-nav `[current]` styling**, and the "de-emphasized" look is genuinely theme-specific (Lumo = filled background + primary text; Aura additionally draws a border via `--vaadin-side-nav-item-*` props). Baking in a single addon default would be wrong for one theme or the other, so it stays app-controlled: the README "Styling → Recipes → Styling root items differently when only a child is active" section now documents it with **separate Lumo and Aura examples** (color/background/border). Reconsider a built-in default only if real users ask for one.

## Cross-version theming

- **Aura theme support (V25). ✅ DONE (`0cde6b5`).** The stylesheet used `--lumo-*` tokens throughout (36 usages, 19 distinct); on V25 the default theme is Aura, where `--lumo-*` are undefined → the addon's own CSS was effectively unstyled. Fixed by making every token a fallback chain (`var(--lumo-…, var(--vaadin-…/--aura-…, <hard>))`) plus the chevron-indicator rework, as written up in [`plans/2026-07-09-aura-theme-support.md`](2026-07-09-aura-theme-support.md). Verified green on both V24 (Lumo) and V25.2.1 (Aura); README compatibility/styling sections updated to match.

## E2E test hardening (do right AFTER the Aura/theming rework)

**Infra-level hardening: ✅ DONE (`14b8cac`).** The `v25` Maven profile (unambiguous version selection), `forceProductionBuild=true` (no more stale-bundle breakage across V24⇄V25 runs), a scopeable `playwright.spec` property, and tightened production-mode timeouts have all landed. What remains below is the deeper **decoupling from `vaadin-popover` internals** — still open, still optional (CI-hygiene, not a release blocker).

Context: the cross-version E2E suite currently couples to Vaadin's **own** `vaadin-popover` internals (overlay location, `role=menu` placement, `positionTarget`/`target`), abstracted behind the dual-form selectors in `e2e/src/test/playwright/lib/popover.ts`. That coupling is the price of E2E-asserting on a framework component that restructured between V24 and V25 — not sloppy test authoring — but a chunk of it is avoidable by giving the addon its own stable, version-independent test seams. Sequenced after the theming rework because that work already operates at the popover boundary.

- **Move behavioural assertions onto addon-owned hooks.** Where a test's real intent is *our* component's behaviour (not framework integration), assert on state the addon itself owns and reflects — existing classes (`.side-nav-rail-popover-header`), attributes (`[root-item]`), ARIA we set — instead of Vaadin's overlay subtree. Candidate: reflect a stable attribute on the rail `SideNavRailItem` when its popover is open (e.g. `data-rail-popover-open`), so "is the popover for item X open?" no longer needs `vaadin-popover-overlay[opened]`. This is partly an **addon** change (add the seam), not only a test change.
- **Quarantine the genuinely framework-dependent tests.** Keep a small, explicitly version-aware set of "popover integration" tests (does an overlay actually open, is `role=menu` present, is slotted content inside it) that legitimately touch `vaadin-popover`. Everything else should stop depending on its internal shape. Goal: shrink `lib/popover.ts` to that small surface.
- **`e2e-v24` split — verdict: not for the crutches.** A second version-pinned E2E module (mirroring `demo`/`demo-v24`) does **not** remove the popover coupling unless specs are forked (which doubles ~113 tests and their maintenance). Its only real win is build robustness: it would retire the `-Dvaadin.version` flip + frontend-wipe dance in `test.sh` and let one `mvn -Pe2e verify` cover both versions. Reconsider later purely as CI hygiene, and only with **shared** specs — after the hook migration above, since that is what actually reduces the pain.
